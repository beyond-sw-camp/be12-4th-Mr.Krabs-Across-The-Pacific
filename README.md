## CI/CD 환경 설정

Mr.Krabs는 5대의 컴퓨팅 쿠버네티스 구성의 복잡성을 줄이기 위해 Ansible을 이용해 쿠버네티스를 설치하기로 하였다.

쿠버네티스 설치 뒤에는 다음과 같이 CI/CD를 수행하기로 하였다.
### 프론트엔드 배포 방식
<!-- 카나리 배포 -->
프론트엔드 프로젝트의 경우 카나리 배포 방식을 사용하기로 하였다.

우리 프로젝트의 프론트엔드는 이미지, 그래프 등 사용자가 직접 눈으로 정보를 확인할 수 있는 요소가 많다.

이것은 개발하거나 유지보수할 기능이 많으며, 기능 변화가 있을 경우 버전이 올라갈 때마다 UI에 대한 기능 변화를 조금씩 바꾸면서 사용자에게 인지시킬 필요가 있다.

따라서 카나리 배포 방식으로 점진적인 기능 향상 및 피드백을 통한 UI 및 UX 개선을 수행하여 접속자 수 변화를 통해 프론트엔드 기능 변화가 미치는 직접적인 영향을 관측하기 쉬운 상황이다.

구체적인 카나리 배포 수행 방식은 새로운 기능을 추가한 버전을 처음에 낮은 빈도로 클라이언트에 제공하고, 점진적으로 새 버전의 노출 빈도를 높여 사용자의 반응을 살피는 구조로 하였다.

이럴 경우 기능 변화에 따른 사용자의 적응 난이도가 낮아지고 작은 기능 변화에도 사용자 이탈에 따른 영향을 즉각적으로 확인하게 된다.

롤링업데이트 방식을 채택하지 않은 것은 해당 방식을 채용할 만큼 프론트엔드가 단순하지 않아, 즉각적인 업데이트를 하면 전체 사용자에게 미치는 영향이 크다.

계획상 개별 버전 업그레이드마다 기능에 격변이 자주 있지 않을 것이므로 블루 그린 배포도 채택하지 않는다.

### 백엔드 배포 방식
<!-- 블루 그린 배포 -->
반면 백엔드 프로젝트는 블루 그린 배포 방식을 따르기로 하였다.

우리 프로젝트의 백엔드는 버전이 바뀔 때마다 API의 내부 작동 방식이 급격하게 바뀌어 왔고, 필요한 정보를 두 종류의 DB에서 받아오는 과정에서 프로젝트 코드의 복잡성이 상당히 증가했다.

앞으로 추가적인 백엔드 개량 및 유지보수가 있을 경우를 가정해도 여전히 기능의 변화가 클 것으로 예측되어, 디버깅 및 배포로 프론트엔드에 전달되는 응답이 크게 바뀔 것으로 예상된다.

또한 현재 GitHub에서 MongoDB에서 그래프 데이터를 받아오는 기능이 있는 브랜치와 없는 브랜치가 기능 측면에서 아직 완전히 병합되지 않은 상태다.

이런 상황에서 롤링업데이트나 카나리 배포로는 요청 및 응답 양식의 변화가 혼란스러울 것으로 예상되어 안정적인 무중단 배포가 불가능한 상황이므로 블루 그린 방식으로 배포하기로 결정하였다.

### CI: CircleCI로 젠킨스가 cURL 요청하여 빌드 및 도커 이미지 푸시까지 일임

우리 프로젝트의 지속 통합 방식은 젠킨스가 CircleCI 서비스에 네트워크 요청을 보내 GitHub 리포지토리에서 푸시가 발생할 경우 CircleCI 측에서 빌드 및 도커 이미지까지 전부 만들고 도커 허브에 올리는 방식이다.

현재 우리 팀이 가진 컴퓨팅 자원이 CD 과정에서 부족할 것으로 예상되어, 젠킨스가 프로젝트 빌드 및 도커 이미지 빌드, 허브 업로드까지 전부 수행하기에 부담이 있는 상황이다.

따라서 CI 과정의 거의 모든 과정을 API 요청을 통해 CircleCI에 일임하여 이런 부담을 해소하고, 젠킨스가 CicleCI로부터 작업 완료를 확인하게 되면 CD 과정으로 넘어가는 방식을 취한다.

### CD: 쿠버네티스에 디플로이먼트 및 스테이트풀 셋 추가

ArgoCD와 같은 다른 수단으로 배포할 경우에는 학습 및 구현에 필요한 노력 대비 효과가 크지 않을 것으로 예상된다.

따라서 CD 과정에서는 젠킨스가 직접 쿠버네티스 마스터로 명령을 보내 프론트 프로젝트, 백엔드 프로젝트, MariaDB RDBMS 및 MongoDB 클러스터를 추가하기로 하였다.

yaml 양식의 데이터를 쿠버네티스 마스터로 전송하여 프론트 프로젝트, 백엔드 프로젝트, MariaDB RDBMS 디플로이먼트를 먼저 구성하되, 대량의 데이터가 저장된 MongoDB는 클러스터로 구성하기로 하였다.

RDBMS는 최대로 확장해도 마스터-슬레이브 구조까지만 구현될 계획이었으므로 조금 복잡한 설정을 감안해도 디플로이먼트로 컨테이너를 올릴 수 있다.

그러나 MongoDB의 경우 이미 MongoDB Atlas를 통해 클러스터를 사용하고 있었기에 해당 구조를 그대로 쿠버네티스 환경에 옮기고자 스테이트풀 셋을 통해 샤드로 분할되는 컨테이너들을 쿠버네티스 상에 배포하기로 한다.
